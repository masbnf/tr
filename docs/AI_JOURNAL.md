# ژورنال کار AI روی این پروژه

ثبت هر سشن کاری Claude روی این ریپو — چه چیزی بررسی شد، چه چیزی ساخته/تغییر کرد، چه چیزی یاد گرفته شد. هدف: شفافیت کامل درباره‌ی اینکه مستندات/کد از کجا آمده‌اند.

---

## ۲۰۲۶-۰۷-۱۲ — ساخت اسکلت اولیه‌ی مستندات زنده (۶ فایل) + تکمیل مجموعه (۱۰ فایل باقی‌مانده)

**درخواست:** ساخت یک سیستم مستندسازی جامع و خودآموز برای پروژه، سنکرون با کد واقعی.

**کاری که انجام شد:**
1. کلون تازه از `master` (کامیت سر: `0c7e99c`) و بررسی کامل ساختار پروژه با `view`/`find`/`grep`.
2. خواندن کامل `git log` (۹ کامیت) شامل پیام‌های کامل کامیت برای استخراج تصمیمات معماری واقعی.
3. اجرای `npm install`, `npm run typecheck`, `npm run lint`, `npm run build` روی sandbox برای تایید وضعیت فعلی — هر سه موفق؛ یک هشدار غیر-بلاک‌کننده کشف شد (bcryptjs در باندل Edge Runtime، مستند شده در `DEBUGGING.md`).
4. استخراج دقیق import graph داخلی با `grep` برای اطمینان از صحت `KNOWLEDGE_GRAPH.md` (نه حدس زدن روابط).
5. کار در دو دور انجام شد چون کاربر ابتدا فقط اسکلت اصلی را خواست: دور اول ۶ فایل (`README`, `PROJECT_MAP`, `ARCHITECTURE`, `MODULES`, `TECHNOLOGIES`, `GLOSSARY`)؛ بعد از تایید کاربر برای «تکمیل مستندات»، ۱۰ فایل باقی‌مانده اضافه شد (`CHANGELOG`, `DECISIONS`, `DEBUGGING`, `ROADMAP`, `KNOWLEDGE_GRAPH`, `CODE_READING_GUIDE`, `LEARNING_PATH`, `FAQ`, `SANDBOX`, `AI_JOURNAL` — این فایل). جمعاً ۱۶ فایل، دقیقاً همان لیست پیشنهادی اولیه‌ی درخواست جامع مستندسازی.
6. اجرای `npm audit` برای بررسی دقیق وضعیت امنیتی وابستگی‌ها (فراتر از هشدار مختصر `npm install`).

**یافته‌های کلیدی که در حین بررسی کشف شد (نه از قبل مشخص بودند):**
- بیشتر صفحات (`/request`, همه‌ی `/admin/*` به‌جز `layout`) عمداً placeholder هستند در حالی‌که لایه‌ی API/دیتابیس پشت‌شان کامل است.
- `SERVICE_HOTSPOTS` (لندینگ، ۱۰ سرویس) با `ServiceType` enum واقعی (۴ مقدار) هماهنگ نیست.
- فونت پروژه سه بار عوض شده (BYekan CDN → Paeez self-hosted → Vazirmatn self-hosted)؛ دستورالعمل فعلی پروژه هنوز BYekan را ترجیحی معرفی می‌کند — ناهماهنگی مستندشده.
- چند فایل یتیم (تصاویر/آرشیو/patch) در ریشه‌ی ریپو باقی مانده‌اند.
- هیچ فایلی `hooks/useAuth.ts`, `hooks/useLocation.ts`, `lib/maps.ts`, `lib/firebase/config.ts` را مصرف نمی‌کند — زیرساخت آماده، منتظر پیاده‌سازی صفحات placeholder.
- `npm audit` روی `next@14.2.5`، ۲۸ آسیب‌پذیری (۱۹ moderate، ۸ high، ۱ critical) نشان می‌دهد که یکی از آن‌ها («Authorization Bypass in Next.js Middleware») مستقیماً به مکانیزم auth ادمین همین پروژه مرتبط است — به همین دلیل در `ROADMAP.md` به بالاترین اولویت منتقل شد.

**تغییری در کد اپلیکیشن داده نشد** — این سشن فقط مستندات اضافه کرد؛ چیزی commit/push نشد (طبق دستورالعمل صریح پروژه).

**درس‌های کلی این سشن:**
- تمام ادعاهای مستندات باید مستقیماً از خروجی ابزار (git log واقعی، grep واقعی، خروجی build واقعی) بیایند، نه از حافظه یا فرض — این باعث شد حداقل ۲ مسئله‌ی واقعی (bcryptjs/Edge warning، ناهماهنگی SERVICE_HOTSPOTS) کشف شود که قبلاً جایی مستند نبودند.
- اجرای واقعی build/lint/typecheck قبل از نوشتن مستندات درباره‌ی «وضعیت فعلی» ضروری بود — پیام دو کامیت اخیر (`f16849a`, `45e33f9`) نگرانی درباره‌ی یک خطای typecheck داشتند که در بررسی مجدد دیگر وجود نداشت؛ بدون تست واقعی این نکته اشتباه مستند می‌شد.

**پیشنهاد برای سشن بعدی:** شروع از یک آیتم مشخص `ROADMAP.md` (پیشنهاد: `request/page.tsx`، طبق [`SANDBOX.md`](./SANDBOX.md#آزمایش-۳-پیاده‌سازی-بخشی-از-فرم-درخواست-advanced))، و بعد از هر تغییر واقعی، یک ورودی جدید اینجا و در `CHANGELOG.md` ثبت شود.
