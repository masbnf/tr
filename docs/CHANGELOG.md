## 2026-07-12 — (اعمال‌شده در sandbox، هنوز commit نشده) تکمیل پنل ادمین (لاگین + داشبورد سفارش‌ها) + رفع یک شکاف امنیتی جدی + رفع یک باگ مخفی در env

**فایل‌ها:** `src/app/admin/login/page.tsx` (پیاده‌سازی کامل)، `src/app/admin/page.tsx` (اتصال به `OrdersTable`)، `src/app/admin/layout.tsx` (افزودن دکمه‌ی خروج، رفع کامنت قدیمی)، `src/components/admin/OrdersTable.tsx` (جدید)، `src/components/admin/LogoutButton.tsx` (جدید)، `src/lib/auth.ts` (افزودن `verifyAdminRequest`)، `src/app/api/requests/route.ts`، `src/app/api/requests/[id]/route.ts` (افزودن auth guard)، `src/app/api/admin/auth/route.ts` (افزودن `DELETE` برای logout)، `.env.example` (مستندسازی escaping)

**دلیل (بخش ۱ — فیچر):** بزرگ‌ترین آیتم باقی‌مانده‌ی «تکمیل پنل ادمین». صفحه‌ی لاگین واقعی (`fetch` به `POST /api/admin/auth`)، و داشبورد سفارش‌ها با جدول واقعی (`GET /api/requests`) + امکان تغییر وضعیت هر سفارش (`PATCH /api/requests/[id]`) با استفاده از `STATUS_CONFIG` موجود برای رنگ/لیبل بج‌ها. دکمه‌ی خروج هم اضافه شد (قبلاً هیچ راهی برای logout نبود) — از طریق یک متد `DELETE` جدید روی همان route فایل لاگین که کوکی را پاک می‌کند.

**دلیل (بخش ۲ — شکاف امنیتی کشف‌شده):** حین ساخت داشبورد مشخص شد `middleware.ts` فقط مسیرهای صفحه (`/admin/:path*`) را محافظت می‌کند، **نه** مسیرهای API (`/api/*`). یعنی `GET /api/requests` (نام/تلفن/موقعیت تمام مشتریان)، `GET /api/requests/[id]`، و `PATCH /api/requests/[id]` (تغییر وضعیت سفارش) کاملاً بدون احراز هویت در دسترس هر کسی با URL بودند — جدی‌تر از CVE بخش قبلی، چون این مستقیماً PII واقعی مشتریان را افشا می‌کرد. یک تابع کمکی `verifyAdminRequest()` به `lib/auth.ts` اضافه شد (همان منطق `middleware.ts`، برای استفاده در Route Handler) و روی هر سه endpoint بالا اعمال شد. `POST /api/requests` عمداً دست‌نخورده ماند (باید عمومی بماند، مطابق `firestore.rules`).

**دلیل (بخش ۳ — باگ env کشف‌شده):** حین تست دستی لاگین با یک هش bcrypt واقعی، لاگین با پسورد درست هم همیشه ۴۰۱ می‌داد. علت: Next.js موقع لود `.env.local`، `$` را syntax گسترش متغیر تفسیر می‌کند و هش bcrypt (که پر از `$` است) را به رشته‌ی خالی تبدیل می‌کند. تایید مستقیم با `@next/env`. راه‌حل: escape کردن `$` به `\$`. `.env.example` به‌روزرسانی شد؛ جزئیات کامل در [`DEBUGGING.md`](./DEBUGGING.md).

**اثر:** پنل ادمین (لاگین + مشاهده/مدیریت سفارش‌ها) کاملاً کارکردی است. یک آسیب‌پذیری real-world (PII افشا) بسته شد. دستورالعمل تولید پسورد ادمین در `.env.example` که قبلاً هرگز کار نمی‌کرد، اصلاح شد.

**تایید:** `typecheck`/`lint`/`build` پاک. تست end-to-end دستی کامل: (۱) `GET /api/requests` بدون کوکی → `401`. (۲) بازدید `/admin` بدون کوکی → ریدایرکت به `/admin/login`. (۳) لاگین با پسورد غلط → `401` با پیام فارسی. (۴) لاگین با پسورد درست (بعد از escape کردن هش) → `200` + کوکی معتبر. (۵) `GET /api/requests` با کوکی معتبر → عبور از auth guard (خطای بعدی‌اش فقط چون این sandbox اعتبارنامه‌ی Firebase ندارد). (۶) بازدید `/admin` با کوکی معتبر → صفحه‌ی داشبورد لود می‌شود، نه ریدایرکت. (۷) `DELETE` برای logout → کوکی پاک می‌شود → `GET /api/requests` دوباره `401`.

**Breaking changes:** ندارد (فقط سخت‌گیرانه‌تر شدن auth روی endpointهایی که از اول قرار بود admin-only باشند).

**⚠️ هنوز باز:** همین شکاف («API routes خارج از پوشش middleware») روی `api/providers/*` و `api/payments` هم صدق می‌کند (هنوز اصلاح نشده — این سشن فقط `api/requests/*` را که داشبورد جدید مستقیماً مصرف می‌کند پوشش داد). به [`ROADMAP.md`](./ROADMAP.md) مراجعه کن.

**وضعیت مستندات:** `README.md`, `ROADMAP.md`, `DEBUGGING.md`, `MODULES.md`, `KNOWLEDGE_GRAPH.md` به‌روزرسانی شدند.

## 2026-07-12 — (اعمال‌شده در sandbox، هنوز commit نشده) پیاده‌سازی فرم ثبت درخواست + آپدیت امنیتی Next.js

**فایل‌ها:** `package.json`, `package-lock.json` (آپدیت `next`/`eslint-config-next` به `14.2.35`)، `src/app/layout.tsx` (افزودن `<Toaster />`)، `src/app/request/page.tsx` (اتصال به `RequestForm`)، `src/components/forms/RequestForm.tsx` (جدید)
**دلیل (امنیت):** `npm audit` نشان داد `next@14.2.5` شامل CVE-2025-29927 (Authorization Bypass in Next.js Middleware) بود که مستقیماً auth ادمین این پروژه را تهدید می‌کرد. آپدیت به `14.2.35` (آخرین patch در همان خط ۱۴.x) این CVE و چند مورد ۲۰۲۵ دیگر را می‌بندد، بدون breaking change.
**دلیل (فرم):** بزرگ‌ترین آیتم «اولویت بالا» در ROADMAP. فرم با `react-hook-form` + `zodResolver(createRequestSchema)` (اسکیمای از‌قبل‌موجود)؛ انتخاب سرویس با گرید دکمه از `SERVICE_OPTIONS`؛ موقعیت با `useLocation` (هوک از‌قبل‌موجود، تا این لحظه بلااستفاده)؛ خطاهای submit با `react-hot-toast` (دیپندنسی از‌قبل نصب‌شده، هرگز wire نشده بود — `<Toaster />` برای همین به `layout.tsx` اضافه شد).
**تصمیم مستندشده:** از ۱۰ اسلاگ `SERVICE_HOTSPOTS`، فقط `battery` و `oil-change` (تطابق یک‌به‌یک بدون ابهام با `ServiceType`) در فرم pre-select می‌شوند؛ بقیه عمداً بدون انتخاب پیش‌فرض می‌مانند چون نگاشت‌شان تصمیم محصولی حل‌نشده است.
**تاثیر:** `/request` دیگر placeholder نیست. آسیب‌پذیری critical بسته شد.
**تایید:** `typecheck`/`lint`/`build` پاک؛ تست دستی `POST /api/requests` با ورودی معتبر (۵۰۰ چون sandbox اعتبارنامه‌ی Firebase ندارد — رفتار درست) و نامعتبر (۴۲۲ با پیام فارسی صحیح).
**⚠️ باقی‌مانده:** `14.2.35` فقط CVEهای ۲۰۲۵ را می‌بندد؛ Next 14 از اکتبر ۲۰۲۵ EOL است و چند آسیب‌پذیری high فقط در ۱۵.۵.۱۸+/۱۶.۲.۶+ رفع شده‌اند — مهاجرت major version یک تصمیم/کار جدا است، به [`ROADMAP.md`](./ROADMAP.md#-اولویت-فوری--امنیت) مراجعه کن.
