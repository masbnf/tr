# تصمیمات معماری (Decision Records)

هر رکورد: مسئله، گزینه‌ها، تصمیم نهایی، دلیل، تاثیر بلندمدت.

## تصمیم: انتخاب فونت فارسی (تاریخچه‌ی سه‌مرحله‌ای)

**مسئله:** رابط کاربری فارسی/RTL به یک فونت خوانا، قابل‌اعتماد، و مطابق برند نیاز دارد.

**مسیر واقعی طی‌شده (از `git log`):**
1. **BYekan از CDN** (کامیت `a5eb886`) — اولین انتخاب، اما CDN غیرقابل‌اعتماد بود و font-face می‌شکست.
2. **Paeez self-hosted** (کامیت `921536b`) — رفع مشکل CDN با self-hosting، اما فونت اسکریپت/تزئینی بود و در سایزهای معمول خیلی کوچک/کم‌خوانا دیده می‌شد (نیاز به افزایش سراسری سایز فونت از ۱۶px به ۱۸px).
3. **Vazirmatn self-hosted** (کامیت `0c7e99c`) — **وضعیت فعلی**. یک فونت variable با پوشش وزن کامل (۱۰۰-۹۰۰)، خواناتر برای متن UI عادی، همچنان self-hosted (بدون وابستگی خارجی).

**تصمیم فعلی:** Vazirmatn، از طریق `next/font/local`، با فایل در `src/app/fonts/Vazirmatn-Variable.woff2`.

**تاثیر بلندمدت:** این پروژه سه بار فونت اصلی را عوض کرده در کمتر از دو هفته — نشان می‌دهد تصمیم فونت باید با احتیاط بیشتری (تست خوانایی در سایزهای واقعی، بررسی اعتبار لایسنس) گرفته شود تا از تغییر مجدد جلوگیری شود. **توجه:** دستورالعمل‌های فعلی این مخزن، «B Yekan / BYekan» را به‌عنوان فونت اصلی ترجیحی معرفی می‌کنند در حالی که کد فعلی از Vazirmatn استفاده می‌کند — این یک ناهماهنگی بین دستورالعمل و وضعیت واقعی کد است که باید آگاهانه حل شود (یا دستورالعمل به‌روز شود، یا فونت دوباره عوض شود؛ با توجه به تاریخچه‌ی بی‌ثباتی فونت، توصیه می‌شود این تغییر فقط با تایید صریح انجام شود، نه خودکار).

## تصمیم: init تنبل (Lazy) برای Firebase Admin SDK

**مسئله:** `next build` روی Vercel هنگام جمع‌آوری اطلاعات مسیرها تمام Route Handlerها را import می‌کند؛ چون `admin.ts` قبلاً در سطح ماژول `cert(...)` را فوراً صدا می‌زد، و Vercel در آن مقطع `FIREBASE_ADMIN_*` را تنظیم نکرده بود، بیلد با خطا شکست («Collecting page data for /api/providers»).

**گزینه‌ها:**
- تنظیم فوری secretها در Vercel قبل از هر بیلد (شکننده — هر بیلد جدید به تنظیم دستی وابسته می‌ماند).
- Lazy-init با Proxy: به‌تعویق‌انداختن اتصال واقعی تا اولین استفاده‌ی runtime.

**تصمیم:** Proxy تنبل (کامیت `c35de6f`) — `adminDb` یک `Proxy` خالی است که فقط در اولین `.collection(...)` واقعی، `getAdminApp()`/`getFirestore()` را صدا می‌زند.

**دلیل:** بیلد باید بدون وابستگی به secretهای اختیاری/محیطی موفق شود؛ خطا فقط باید در runtime واقعی (وقتی endpoint واقعاً صدا زده می‌شود) رخ دهد، نه در زمان بیلد.

**تاثیر بلندمدت:** این الگو باید برای **هر** سرویس بیرونی جدید با env var اختیاری تکرار شود (قانونی که در [`ARCHITECTURE.md`](./ARCHITECTURE.md#الگوی-lazy-init-برای-firebase-admin) و دستورالعمل‌های پروژه هم تاکید شده: «Keep Firebase and deployment initialization compatible with build-time and serverless environments»).

## تصمیم: `jose` به‌جای `jsonwebtoken`

**مسئله:** `middleware.ts` روی Edge Runtime اجرا می‌شود که Node.js API کامل ندارد.

**گزینه‌ها:** `jsonwebtoken` (استاندارد رایج Node، اما متکی به ماژول‌های Node مثل `crypto` کلاسیک) در برابر `jose` (متکی به Web Crypto API، edge-safe).

**تصمیم:** `jose`.

**تاثیر بلندمدت:** هر تابع رمزنگاری/JWT جدیدی که در مسیر اجرای `middleware.ts` قرار می‌گیرد باید edge-compatible باشد. (نکته‌ی مرتبط: `bcryptjs` که در همان فایل `lib/auth.ts` است Edge-safe **نیست** — به [`DEBUGGING.md`](./DEBUGGING.md#هشدار-بیلد-bcryptjs-در-باندل-edge-runtime) مراجعه کن برای این تنش شناخته‌شده.)

## تصمیم: `bcryptjs` (pure-JS) به‌جای `bcrypt` نیتیو

**مسئله:** هش/مقایسه‌ی پسورد ادمین در محیط build/serverless که کامپایل native module می‌تواند شکننده باشد.

**تصمیم:** `bcryptjs`، بدون وابستگی به native binding.

**دلیل:** هم‌راستا با فلسفه‌ی «سازگاری کامل با serverless/بدون کامپایل» که در انتخاب `jose` هم دیده می‌شود.

**تاثیر شناخته‌شده:** چون `bcryptjs` هنوز به چند Node API (`setImmediate`, `process.nextTick`) متکی است، وقتی از طریق `lib/auth.ts` به باندل Edge Runtime (`middleware.ts`) کشیده می‌شود، هشدار build تولید می‌کند (نه خطای بلاک‌کننده). جزئیات در `DEBUGGING.md`.

## تصمیم: دو SDK جدا برای Firebase (Client + Admin)

**مسئله:** هم نیاز به دسترسی کامل/امن سمت سرور (برای Route Handlerها) و هم زیرساخت احتمالی برای فیچر real-time سمت کلاینت.

**تصمیم:** `firebase` (Client SDK) در `lib/firebase/config.ts` برای آینده، `firebase-admin` در `lib/firebase/admin.ts` برای همه‌ی عملیات فعلی.

**تاثیر:** کنترل دسترسی واقعی در Route Handlerها پیاده می‌شود، نه در Firestore Rules (که همه‌چیز غیر از create روی `requests` را می‌بندد). این یعنی **هر endpoint جدید مسئول چک auth خودش است** — Rules محافظ دوم نیست، محافظ اول Route Handler است.

## تصمیم: داده‌ی UI محلی به‌جای مرکزی برای محتوای صرفاً نمایشی

**مسئله:** لندینگ صفحه به آرایه‌های داده‌ای زیادی نیاز دارد (`STEPS`, `STATS`, `SERVICE_HOTSPOTS`, `MOCK_STATS`, ...).

**تصمیم (الگوی مشاهده‌شده، نه یک کامیت خاص):** داده‌ی UI-only که فقط یک کامپوننت مصرف می‌کند، همان‌جا به‌عنوان ثابت محلی تعریف می‌شود؛ فقط داده‌ای که business logic/دیتابیس هم به آن نیاز دارد (enum، status config) به `src/constants/` می‌رود.

**تاثیر:** این یعنی `SERVICE_HOTSPOTS` (لندینگ) و `SERVICE_OPTIONS` (فرم واقعی، در `constants/`) دو منبع داده‌ی جدا برای مفهوم «نوع خدمت» هستند و می‌توانند از هم واگرا شوند (که همین الان هم شده‌اند) — این ریسک ذاتیِ همین الگوی طراحی است، نه یک باگ گذرا.

## تصمیم: بدون CI/CD جداگانه؛ Vercel به‌تنهایی مسئول بیلد/دیپلوی

**مسئله:** نیاز به یک مسیر ساده برای دیپلوی یک اپ کوچک تک‌تیمی.

**تصمیم:** بدون GitHub Actions یا pipeline جدا؛ اتصال مستقیم Vercel↔GitHub، دیپلوی خودکار از `master`.

**تاثیر بلندمدت:** هیچ safety net خودکاری (مثل اجرای تست/lint در CI قبل از merge) وجود ندارد؛ مسئولیت اجرای `npm run lint`/`npm run typecheck`/`npm run build` قبل از push کاملاً روی توسعه‌دهنده (یا ابزار کمکی مثل Claude) است. اگر تیم بزرگ‌تر شود، افزودن یک GitHub Action ساده برای این سه دستور قبل از merge توصیه می‌شود.
